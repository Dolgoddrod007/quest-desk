Нефункциональные требования

IDНазваниеОписание требованияПриоритетОбоснование
NFR-001Хеширование паролейПароли пользователя при регистрации и изменении должны хешироваться с использованием bcrypt с коэффициентом стоимости не менее 10. Хешированное значение сохраняется в базе данных.MUST HAVEХранение паролей в открытом виде или слабом хеше приведет к компрометации учетных записей при утечке данных.
NFR-002Шифрование трафикаВсе данные, передаваемые между клиентом и сервером (включая страницы, API-запросы, формы), должны шифроваться по протоколу HTTPS с использованием TLS версии не ниже 1.2.MUST HAVEПерехват трафика позволяет злоумышленнику получить пароль, email и другие личные данные. Требуется законом о защите персональных данных.
NFR-003Время отклика при типовой нагрузкеПри одновременной работе 10 пользователей время ответа сервера (от получения запроса до отправки первого байта) для операции создания партии, загрузки листа персонажа и создания квеста не должно превышать 2 секунды в 95% случаев.MUST HAVEИсследования показывают, что задержка более 2 секунд снижает удовлетворенность пользователей. Для MVP ожидается не более 10 одновременных пользователей.
NFR-004Пропускная способность API (RPS)API-сервер должен выдерживать пиковую нагрузку 50 запросов в секунду (RPS) при выполнении операции чтения (получения списка партий, данных пользователей) без увеличения времени ответа более чем до 3 секунд для 95% запросов.SHOULD HAVE50 RPS достаточно для обслуживания до 1000 активных пользователей при средней частоте запросов 3 в минуту. Это создаёт запас на случай роста популярности.
NFR-005Объем передаваемых данныхОбъем данных, передаваемых при загрузке страницы персонажа (включая HTML, CSS, JS и JSON-ответ), не должен превышать 250 КБ после сжатия.SHOULD HAVEПользователи могут подключаться через мобильную сеть с ограниченной пропускной способностью. Меньший объем ускоряет и экономит трафик.
NFR-006Адаптивность для мобильных устройствНа устройствах с экраном шириной 375 пикселей все страницы должны отображаться без горизонтальной прокрутки; кнопки и интерактивные элементы должны иметь размер не менее 44×44 пикселей.MUST HAVE70% игроков используют телефон за столом для доступа к листам персонажей. Отсутствие адаптации сделает продукт неудобным.
NFR-007Локализация интерфейсаИнтерфейс должен поддерживать русский язык как основной; все надписи, сообщения должны быть переведены на русский.MUST HAVEЦелевая аудитория - русскоязычные игроки.
NFR-008Атомарность операцийСистема должна обеспечивать атомарность операций создания и обоснования ключевых сущностей (персонаж, квест, NPC): при сбое (обрыв соединения, падение сервера) данные либо полностью сохраняются, либо возвращаются к состоянию до операции; частичное сохранение невозможно.MUST HAVEПотеря части данных приведет к некорректному состоянию и недовольству пользователей.
NFR-009Время восстановления после сбояВремя восстановления после сбоя не должно превышать 5 минут при условии отсутствия аппаратных сбоев.COULD HAVEДля MVP допустимы кратковременные перерывы, но важно быстро вернуть систему к строю.
NFR-010Модульность кода бэкендаИсходный код бэкенда должен быть организован в модули по функциональному признаку. Каждый модуль должен содержать модели, контроллеры и сервисы только своей области.SHOULD HAVEМодульность упрощает внесение изменений и отладку.
NFR-011Поддержка браузеровВеб-интерфейс должен корректно работать в последних версиях браузеров: Google Chrome (версия 120+), Mozilla Firefox (версия 115+), Apple Safari (версия 17+), Microsoft Edge (версия 120+).MUST HAVEЭти браузеры покрывают более 95% целевой аудитории. Поддержка устаревших браузеров не требуется.
NFR-012Логирование событийСистема должна логировать следующие события: регистрацию нового пользователя, вход в систему (успешный и неуспешный), создание партии, удаление партии, создание персонажа, удаление персонажа, создание квеста, изменение квеста. Каждая запись лога должна содержать временную метку (UTC), идентификатор пользователя и результат (успех/неуспех).COULD HAVEЛоги помогают отслеживать ошибки и собирать статистику для KPI. В MVP это не критично.
NFR-013Емкость храненияБаза данных должна поддерживать хранение до 100 партий, 10 000 персонажей, 20 000 квестов без потери производительности запросов (время выполнения типовых запросов не должно превышать 1 секунду).COULD HAVEТакой объем запросов покрывает потребности MVP. Требование гарантирует, что архитектура не ограничит рост.
NFR-014Горизонтальное масштабированиеПриложение должно поддерживать возможность горизонтального масштабирования путем добавления дополнительных экземпляров бэкенда без изменений исходного кода (архитектура предполагает stateless-серверы).WONT HAVEДля MVP с низкой нагрузкой масштабирования не требуется, это усложнит разработку. Реализация отложена до следующих версий.
NFR-015Политика конфиденциальностиНа сайте должна быть доступна страница с Политикой конфиденциальности, содержащая информацию о том, какие персональные данные собираются, как они обрабатываются и хранятся, а также о правах пользователей.COULD HAVEТребуется для соответствия законодательству о персональных данных (152-ФЗ). В MVP достаточно базовой версии.
NFR-016Согласие на обработку данныхПри регистрации пользователь должен явно подтвердить согласие на обработку персональных данных (чекбокс «Я согласен с Политикой конфиденциальности»). Без этого согласия регистрация невозможна.COULD HAVEЮридически значимое действие, необходимое для легитимной обработки данных.
