# Обоснование выбора технологии хранения данных

Приведено детальное обоснование выбора реляционных баз данных (PostgreSQL) для всех основных сущностей системы, а также дополнительных инструментов (кэш, очереди). Анализ выполнен по каждой сущности согласно шагам, описанным в методике.

## Сущности и требования к хранению

### 1. users
**Хранимые данные:** идентификатор, email, имя пользователя, пароль, роль, ссылка на аватар, флаги активности и привилегий, дата создания, дата присоединения, дата последнего входа.  
**Характер данных:** транзакционные.  
**Ожидаемый объём:** 1 ГБ – 1 ТБ.  
**Как система пишет и читает данные?** OLTP + Key-Value lookup (поиск по ключу, email, username).  
**Насколько критична немедленная консистентность?** Strong Consistency (права и профиль должны быть согласованы мгновенно). Для кеша допустима Eventual Consistency.  
**Может ли система быть недоступна?** Доступность 99.9% (допустимы короткие перерывы).  
**Часто ли меняется структура данных?** Жёсткая схема (набор атрибутов фиксирован).  
**Нужны ли транзакции между несколькими объектами?** Да, критично (регистрация, смена прав).  
**Какие запросы нужно выполнять?** По ключу, фильтрация по роли/статусу.  
**Какой бюджет?** Дёшево: Open-source.  
**Итоговое решение:** реляционная БД (PostgreSQL) + кэш в оперативной памяти (например, Redis).

## Общий вывод

Выбрана реляционная база данных PostgreSQL как единое хранилище, дополненное кэшем (Redis) и очередью задач для асинхронной отправки уведомлений. PostgreSQL обеспечивает:
- Полноценную поддержку ACID-транзакций для согласованности;
- Гибкость за счёт типа JSONB для варьирующихся структур персонажей и NPC;
- Высокую производительность при правильном индексировании;
- Бесплатную лицензию (Open-source);
- Возможность партиционирования для логов.
